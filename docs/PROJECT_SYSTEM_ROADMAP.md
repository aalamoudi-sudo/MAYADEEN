# KAG Project Management System Roadmap

## الهدف

تحويل لوحة KAG الحالية من ملف HTML تنفيذي إلى نظام حقيقي لإدارة المشروع، يعتمد على بيانات موثوقة، صلاحيات، سجل تغييرات، ومتابعة يومية قابلة للتدقيق.

## المرحلة 1: تثبيت الثقة في البيانات

- إضافة أعمدة إلزامية في مصدر البيانات: `status`, `percent_complete`, `actual_start`, `actual_finish`, `baseline_start`, `baseline_finish`, `last_update`, `updated_by`.
- عدم اعتبار المهمة مكتملة إلا إذا كانت حالتها في المصدر `مكتملة` أو لديها `actual_finish`.
- فصل الحالة الزمنية عن حالة الإنجاز:
  - حالة الإنجاز: لم تبدأ، قيد التنفيذ، مكتملة، معلقة.
  - حالة الجدول: ضمن الخطة، مستحقة قريباً، متأخرة، حرجة.
- إضافة قواعد تحقق قبل عرض المؤشرات: تاريخ بداية مفقود، تاريخ نهاية قبل البداية، مسؤول مفقود، حالة غير معروفة.
- إظهار "جودة البيانات" كتنبيه إداري واضح، وليس كعنصر ثانوي.

## المرحلة 2: تحويل Google Sheet إلى مصدر مضبوط

- إنشاء قاموس أعمدة ثابت وموثق للشيت.
- قفل الأعمدة الحسابية وحماية صيغ الحالة.
- استخدام Apps Script كواجهة API منظمة لا تعيد كامل الشيت فقط، بل endpoints مثل:
  - `/tasks`
  - `/milestones`
  - `/risks`
  - `/decisions`
  - `/updates`
- إضافة نسخة تاريخية من كل تحديث في sheet منفصل باسم `Audit Log`.
- إضافة `row_id` ثابت لكل مهمة حتى لا تعتمد العلاقات على الاسم أو ترتيب الصفوف.

## المرحلة 3: صلاحيات واعتماد

- استبدال كلمة المرور الموجودة داخل HTML بتسجيل دخول حقيقي.
- تحديد أدوار:
  - Executive: قراءة المؤشرات والتقارير.
  - Project Manager: تعديل الحالة، التصعيد، الاعتمادات.
  - Workstream Owner: تحديث مهامه فقط.
  - Viewer: قراءة فقط.
- ربط كل تحديث بمستخدم وتاريخ وسبب مختصر.
- منع تعديل تاريخ الأساس baseline إلا بصلاحية مدير المشروع.

## المرحلة 4: دورة عمل تشغيلية

- إضافة شاشة "تحديث يومي" لكل مسؤول تعرض مهامه فقط.
- إضافة نموذج تحديث سريع: الحالة، نسبة الإنجاز، العائق، الإجراء القادم، تاريخ التحديث.
- إضافة workflow للتصعيد:
  - عائق جديد.
  - قرار مطلوب.
  - اعتماد مطلوب.
  - خطر مستجد.
- إضافة SLA للتحديث: أي مهمة قيد التنفيذ دون تحديث خلال 3 أيام تظهر كـ "تحتاج تحديث".

## المرحلة 5: المخاطر والقرارات كبيانات حقيقية

- فصل سجل المخاطر عن استنتاجات التأخير.
- حقول المخاطر المقترحة: `risk_id`, `title`, `category`, `probability`, `impact`, `owner`, `treatment`, `due_date`, `status`, `linked_task_id`.
- حقول القرارات المقترحة: `decision_id`, `title`, `requester`, `approver`, `priority`, `due_date`, `status`, `impact`, `linked_risk_id`.
- ربط المخاطر والقرارات بالمهام عبر IDs لا عبر نصوص الأسماء.

## المرحلة 6: تقارير تنفيذية

- تقرير أسبوعي PDF تلقائي:
  - نسبة الإنجاز الفعلية.
  - الانحراف عن الخطة.
  - أهم 10 مهام حرجة.
  - المخاطر المفتوحة.
  - القرارات المطلوبة من المالك.
- لقطة يومية محفوظة للمؤشرات حتى يمكن مقارنة الأداء عبر الزمن.
- خط أساس baseline للمقارنة بدل الاعتماد على تواريخ الخطة الحالية فقط.

## المرحلة 7: بنية تقنية قابلة للنمو

- فصل النظام إلى:
  - Frontend: React أو Vue.
  - Backend/API: Node.js أو Python FastAPI.
  - Database: PostgreSQL أو Supabase.
  - Auth: Google Workspace أو Microsoft Entra.
- إبقاء نسخة HTML الحالية كنموذج أولي، ثم نقل المنطق تدريجياً إلى تطبيق حقيقي.
- إضافة اختبارات لحساب الحالة، التأخير، نسب الإنجاز، وجودة البيانات.

## أولويات التنفيذ المقترحة

1. إصلاح منطق الحالة والمزامنة في النسخة الحالية.
2. توحيد أعمدة Google Sheet وإضافة حقول الإنجاز الفعلي.
3. بناء Audit Log للتحديثات.
4. إنشاء شاشة تحديث يومي للمسؤولين.
5. نقل المخاطر والقرارات إلى جداول مستقلة.
6. إضافة صلاحيات وتسجيل دخول حقيقي.
7. تحويل النموذج إلى تطبيق ويب بقاعدة بيانات.
