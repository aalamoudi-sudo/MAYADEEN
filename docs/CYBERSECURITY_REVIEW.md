# KAG Dashboard Cybersecurity Review

## الخلاصة

تم إنشاء نسخة آمنة مخصصة للرفع العام على GitHub باسم `KAG_Dashboard_GitHub_Safe.html`. النسخة الداخلية `KAG_Dashboard_Final.html` تعمل للعرض الداخلي وقد تحتوي على بيانات مشروع مدمجة، لذلك لا ترفعها إلى مستودع عام إذا كانت البيانات غير مخصصة للنشر.

## النتيجة الحالية

- مستوى المخاطر للنسخة الداخلية: متوسط، بشرط عدم رفع بيانات المشروع لمستودع عام.
- مستوى المخاطر للنسخة العامة `KAG_Dashboard_GitHub_Safe.html`: منخفض كواجهة static، بشرط أن يكون Google Apps Script محمياً ولا يحتوي الرابط على صلاحيات كتابة عامة.

## ما تم تحسينه الآن

- تقليل خطر XSS في نوافذ التفاصيل والمرحلة.
- تعقيم مخرجات مهمة في صفحات المعالم، المسؤولين، المهام العاجلة، المخاطر، خطط المعالجة، والقرارات.
- إصلاح استخدام رابط API المحفوظ في `localStorage`.
- إصلاح عمل التحديث التلقائي بعد تسجيل الدخول.
- منع حساب المهمة المنتهية كمكتملة تلقائياً، وهذا يحسن سلامة المؤشرات التشغيلية.
- إزالة كلمة المرور الثابتة من منطق الدخول.
- إزالة رابط Google Apps Script الافتراضي.
- تعطيل أي اتصال خارجي تلقائي ما لم يدخل المستخدم رابط Apps Script صالحاً.
- إزالة CDN الخارجي لـ Chart.js وGoogle Fonts من النسخة الحالية.
- إضافة CSP أساسي مناسب لصفحة static.
- إنشاء نسخة GitHub عامة بدون بيانات WBS مدمجة.

## المخاطر الحرجة المتبقية

### 1. كلمة المرور داخل ملف HTML

تمت إزالة كلمة المرور الثابتة. شاشة الدخول الحالية بوابة عرض فقط وليست مصادقة حقيقية.

الإجراء المطلوب:
- عدم استخدام هذا الأسلوب في الإنتاج.
- نقل المصادقة إلى خادم backend أو استخدام Google Workspace / Microsoft Entra.
- جعل الصفحة تتلقى session/token مؤقتاً بدلاً من كلمة مرور ثابتة.

### 2. رابط Google Apps Script

تمت إزالة الرابط الافتراضي من الواجهة. أي رابط يدخله المستخدم سيبقى في `localStorage` على جهازه فقط، لكن أمان البيانات يعتمد على صلاحيات Apps Script نفسه.

الإجراء المطلوب:
- ضبط Apps Script ليتحقق من هوية المستخدم.
- عدم السماح بالكتابة دون صلاحية.
- فصل read-only endpoint عن update endpoint.
- تدوير الرابط إذا كان قد تم تداوله خارج الفريق.

### 3. الاعتماد على CDN خارجي

تمت إزالة تحميل Chart.js وGoogle Fonts من CDN. الرسوم تستخدم fallback داخلياً.

الإجراء المطلوب:
- استضافة Chart.js محلياً داخل المشروع، أو إضافة SRI و`crossorigin`.
- تثبيت النسخة وعدم الاعتماد على CDN في بيئات حساسة.

### 4. سياسة CSP

تمت إضافة CSP أساسي. ما زالت الصفحة تستخدم inline scripts وinline event handlers، لذلك السياسة ليست بأقصى صرامة ممكنة. لتقويتها أكثر يجب نقل JavaScript إلى ملفات منفصلة واستبدال event handlers.

الإجراء المطلوب:
- نقل JavaScript إلى ملف منفصل.
- استبدال `onclick` و`onmouseover` بـ `addEventListener`.
- إضافة CSP تمنع السكربتات غير المصرح بها.

### 5. لا يوجد سجل تدقيق حقيقي

لا يوجد Audit Log يثبت من عدل ماذا ومتى. هذا مهم جداً لنظام إدارة مشروع حقيقي.

الإجراء المطلوب:
- إضافة Audit Log في Google Sheet أو قاعدة بيانات.
- تسجيل المستخدم، الوقت، الحقل القديم، الحقل الجديد، وسبب التعديل.

## شروط النشر الآمن

لنشر عام على GitHub، استخدم `KAG_Dashboard_GitHub_Safe.html` فقط، وتأكد من التالي:

1. لا ترفع `KAG_Dashboard_Final.html` إلى مستودع عام إذا كانت بيانات WBS حساسة.
2. حماية API بمصادقة وصلاحيات.
3. تدوير رابط Apps Script القديم إذا كان قد انكشف.
4. عدم منح Apps Script صلاحيات كتابة عامة.
5. إضافة Audit Log وصلاحيات حسب الدور عند التحول لنظام إنتاجي.

## التوصية

استخدم النسخة الحالية كـ prototype داخلي فقط. للتحول إلى نظام حقيقي، يجب بناء backend بسيط بين الواجهة وGoogle Sheet أو قاعدة البيانات، بحيث تكون المصادقة والصلاحيات والتحقق من البيانات خارج ملف HTML.
